> 판단 기준: **"상대방의 응답이 지금 당장 필요한가?"**
> - 필요하다 → REST (동기) — 전화처럼 바로 응답
> - 아니다 → Redis Streams (비동기) — 카톡처럼 나중에 확인

## 시나리오별 통신 흐름

**1. 유저가 메타 덱 페이지를 연다**

```
유저 → API Gateway ──REST──→ Content Service ──→ Redis 캐시 → 덱 목록 반환
```

유저가 화면 앞에서 기다리고 있으므로 REST.

**2. 유저가 소환사 전적을 검색한다**

```
유저 → API Gateway ──REST──→ Content Service ──REST──→ Riot API → 전적 반환
```

검색한 소환사가 DB에 없는 신규 유저라면:

```
Content ──Redis Streams──→ Pipeline ("이 소환사 매치 수집해줘")
```

유저에게는 Riot API로 바로 전적을 보여줌. 매치 수집은 백그라운드에서 나중에 처리하면 되므로 Redis Streams.

**3. 매치 데이터 수집 → 통계 집계 (배치)**

```
Pipeline 내부에서 수집 → 집계 수행 (같은 서비스)
    ↓
Pipeline ──REST──→ Content ("새 덱 통계 여기 있어, 저장해")
    ↓
Pipeline ──Redis Streams──→ Content ("통계 갱신됐어, 캐시 비워")
```

- 수집과 집계가 같은 서비스이므로 내부 호출로 처리
- 결과 전달: Content가 받아서 저장해야 하므로 REST
- 캐시 무효화: Content가 즉시 안 비워도 됨 → Redis Streams

**4. 유저가 매치 분석을 요청한다 (유료)**

```
유저 → API Gateway ──REST──→ Pipeline Service ──→ 자체 매치 데이터 분석 → 결과 반환
```

Pipeline이 이미 매치 데이터를 갖고 있으므로 서비스 간 호출 없이 자체 분석.

**5. 클라이언트 프로그램이 인게임 데이터를 전송한다**

```
클라이언트 프로그램 ──REST──→ API Gateway ──REST──→ Pipeline Service → DB 저장
```

인게임 중 실시간으로 라운드별 데이터를 전송. 매치 종료 후 Riot API 데이터와 매칭하여 AI 분석에 활용.

**6. 새 패치가 감지된다**

```
Pipeline ──Redis Streams──→ Content ("새 패치 14.22 나왔어, 이전 데이터 아카이브해")
```

패치 전환은 즉시 처리 안 해도 됨 → Redis Streams.

## 통신 요약

| 통신 경로 | 방식 | 시나리오 |
| --- | --- | --- |
| API Gateway → 각 서비스 | REST | 모든 유저 요청 |
| Content → Pipeline | Redis Streams | #2 미수집 소환사 수집 요청 |
| Pipeline → Content | REST | #3 집계 결과 전달 |
| 클라이언트 프로그램 → Pipeline | REST | #5 인게임 데이터 실시간 전송 |
| Pipeline → Content | Redis Streams | #3 캐시 무효화, #6 패치 전환 알림 |

## 이벤트 스트림 (Redis Streams)

| 스트림 | 발행자 | 구독자 | 용도 |
| --- | --- | --- | --- |

| `stats.updated` | Pipeline | Content | 캐시 무효화 |
| `patch.detected` | Pipeline | Content | 패치 전환 알림 |
