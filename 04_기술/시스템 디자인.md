## 서비스 분리

2인 팀 규모에 맞춰 3개 서비스 + API Gateway로 구성. 코드 레벨에서 모듈 경계를 명확히 하여 트래픽 증가 시 서비스 분리 가능하도록 설계.

```
[Client (React)]    [클라이언트 프로그램 (데스크톱)]
       │                        │
       ▼                        │ 인게임 데이터 실시간 전송
[API Gateway] ─── 라우팅, 인증 검증, Rate Limiting
       │                        │
       ├─→ [Content Service]    │  메타 덱, 소환사 검색, 매치 히스토리
       ├─→ [Pipeline Service] ←─┘  매치 수집 + 분석 (유료)
       └─→ [Auth Service]         회원가입, 로그인, RSO 연동, 결제

[Pipeline Service]  Riot API 폴링, 매치 수집, 클라이언트 데이터 수신, 통계 분석
       │
       └─→ [Content Service]  집계 결과 전달, 캐시 갱신
```

### 서비스별 역할

| 서비스 | 역할 | 스케일링 특성 |
| --- | --- | --- |
| API Gateway | 라우팅, JWT 검증, Rate Limiting | 트래픽 비례 수평 확장 |
| Content Service | 메타 덱/꿀 덱 조회, 소환사 프로필, 매치 히스토리 | 읽기 위주, 캐싱 효과 큼 |
| Pipeline Service | Riot API 폴링, 매치 수집, 클라이언트 인게임 데이터 수신/저장, 덱 클러스터링, 통계 집계, 매치 분석 (클라이언트 + API 데이터 통합), 매치 컨텍스트 분석, 카테고리별 등급 산출 (S/A/B/C) | 수집은 I/O 바운드, 분석은 CPU 바운드 |
| Auth Service | 회원/인증/결제 | 낮은 트래픽, 높은 안정성 |

### 서비스 분리 근거

| 분리 | 근거 |
| --- | --- |
| Content ↔ Pipeline | Content는 읽기/캐싱 위주, Pipeline은 수집/분석 — 장애 격리 및 스케일링 패턴이 다름 |
| Auth 독립 | 보안 민감, 변경 주기가 다름, 결제 로직 격리 |

## 매치 분석

클라이언트 프로그램의 인게임 실시간 데이터와 Riot API의 매치 결과를 통합하여 분석한다. 라운드별 운영 판단(골드 관리, 레벨업 타이밍, 리롤 효율)부터 최종 보드 상태(아이템, 증강, 구성)까지 전 과정을 메타 덱 통계와 비교하여 피드백을 생성한다.

| 분석 항목 | 방법 |
| --- | --- |
| 덱 구성 | 유저 덱 vs 같은 덱 상위 등수 구성 비교 |
| 아이템 | 챔피언별 최적 아이템 vs 실제 장착 아이템 |
| 증강 | 해당 덱에서 증강별 평균 등수 비교 |
| 특성 활성화 | 비효율적 특성 활성 감지 |
| 레벨/경제 | 최종 레벨, 남은 골드, 탈락 시점을 상위권과 비교 |

> 클라이언트 미실행 매치는 분석 불가. 클라이언트 프로그램을 실행한 매치만 AI 리포트를 생성할 수 있다.

### 분석 응답 구조

분석 결과는 다음 구조로 저장/반환한다.

- **종합 등급**: S/A/B/C
- **카테고리별 등급**: 운영, 덱 선택, 아이템, 증강 각각 S/A/B/C + 설명
- **매치 컨텍스트**: 로비 태그, 8명 평균 행동 패턴, 플레이어 판단 평가
- **상세 분석**: 카테고리별 근거 텍스트 (레벨업 타이밍, 골드 관리, 아이템 비교, 증강 평가 등)

Code(결정론적 분석)가 등급 산출과 통계 비교를 담당하고, Claude API가 컨텍스트 분석과 자연어 설명을 생성한다.

## DB 전략

서비스별 독립 MySQL 인스턴스 운영 (Database per Service). 서비스 간 DB 직접 접근 없이, API를 통해서만 데이터를 주고받는다.

| 서비스 | DB | 선택 이유 |
| --- | --- | --- |
| Content | MySQL + Redis | 덱/가이드 데이터 + 소환사 캐싱 + 즐겨찾기/검색 이력 |
| Pipeline | MySQL | 매치 원본, 클라이언트 수집 데이터, 집계 결과, 분석 결과 |
| Auth | MySQL | 유저, 세션, 결제 내역 |

> 같은 EC2 위에서 Docker로 MySQL 컨테이너를 서비스별로 분리 운영. Pipeline의 배치 집계 쿼리가 Content/Auth에 영향을 주지 않도록 물리적으로 격리.
