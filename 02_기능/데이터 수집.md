## 수집 목적

| 기능 | 필요한 데이터 | 수집 방식 |
| --- | --- | --- |
| 메타 덱 | 대량의 매치 데이터 (덱 클러스터링, 통계 계산) | 사전 수집 필수 — 유저 방문과 무관하게 데이터가 쌓여 있어야 통계 산출 가능 |
| 전적 검색 | 특정 유저의 매치 히스토리 | 실시간 조회 — 유저 검색 시 Riot API 직접 호출 |
| AI 분석 리포트 | 해당 매치 + 비교용 통계 | 메타 덱 통계가 이미 있으면 비교 가능 |

## 수집 전략

Snowball 크롤링: 고티어 시드 플레이어의 매치를 수집하고, 매치 참가자를 새 시드로 추가하며 커버리지를 확장하는 방식. TFT 매치 데이터의 공개 데이터셋이 없으므로 Riot API 크롤링이 유일한 방법이다.

```
League API → 시드 플레이어 수집 (Challenger/GM/Master)
    ↓
Match History API → 매치 ID 수집
    ↓
Match Detail API → 매치 상세 데이터 수집 (1건당 8명 데이터)
    ↓
매치 참가자 → Snowball 확장 (Platinum 이상만 시드 추가)
    ↓
DB 적재 → 통계 집계
```

## 수집 대상

| 항목 | 기준 | 비고 |
| --- | --- | --- |
| 티어 | Platinum 이상 | Platinum/Diamond은 Snowball로 자연 확보 (League API 직접 조회 안 함) |
| 리전 | KR 우선 | 이후 글로벌 확장 |
| 게임 모드 | 전체 수집 | 통계 집계 시 모드별 분리 |

## 데이터 정책

### 패치 전환

- 새 패치 감지 시 이전 패치 데이터 수집은 계속 (진행 중인 매치 마무리)
- 새 패치 데이터는 별도 집계, 기존 패치 통계는 아카이브
- 새 패치 데이터 1,000 매치 미만이면 "데이터 수집 중" 표시

### 데이터 보존

| 데이터 | 보존 기간 | 근거 |
| --- | --- | --- |
| 매치 원본 데이터 | 현재 패치 + 직전 2패치 | 패치별 통계 추이 비교에 필요 |
| 집계 통계 (덱 순방률 등) | 전체 시즌 | 용량 작음, 추이 분석용 |
| 시드 풀 | 상시 유지 | 비활성 시드는 주기적 정리 |

### 최소 데이터 기준

| 상황 | 기준 | 처리 |
| --- | --- | --- |
| 통계 유의미 최소치 | 10,000 매치 | 런칭 전 최소 확보 목표 |
| 덱 티어 부여 최소 샘플 | 50 매치 | 미만이면 "데이터 부족" 표시 |
| 패치 초반 완화 | 20 매치 | 전체 1,000 매치 미만일 때 |

## 초기 부트스트랩

런칭 전 최소 데이터 확보 전략:

- **목표**: 최소 10,000 매치
- **방법**: 시드 풀(Challenger/GM/Master)의 최근 매치를 일괄 수집 → 유니크 매치 상세 수집
- **완료 기준**: 주요 덱 클러스터가 10개 이상, 각 클러스터 최소 100 매치 이상

## 클라이언트 데이터 수집 (1차)

Riot API만으로는 수집할 수 없는 심층 데이터를 위한 클라이언트 프로그램. 오버레이 기능은 제공하지 않으며, 순수 데이터 수집 목적.

**수집 데이터 (API 미제공):**
- 라운드별 골드/경험치 변화
- 리롤 횟수 및 타이밍
- 아이템 조합 순서
- 스테이지별 보드 스냅샷
- 전투 라운드별 데미지 로그
- 증강 선택지 (선택하지 않은 것 포함)

**활용:**
- AI 피드백: 수집된 데이터 기반으로 플레이 패턴 분석 및 개선점 제안
- 경쟁사와의 차별점 — 다른 사이트는 AI 분석이 없으나, 풍부한 클라이언트 데이터 + AI 피드백으로 차별화
- 플레이어 분석 레이더 차트의 정밀도 향상

> 수집 파이프라인 구현 상세(폴링 주기, API Key 전략, Rate Limit 관리, 에러 처리)는 기술 문서 참고. API 스펙은 [[Riot TFT API]] 참고.
