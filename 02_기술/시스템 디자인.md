## 서비스 분리

2인 팀 규모에 맞춰 6개 서비스 + API Gateway로 구성.

```
[Client (React)]
       │
       ▼
[API Gateway] ─── 라우팅, 인증 검증, Rate Limiting
       │
       ├─→ [Comp Service]       추천 덱, 꿀 덱, 큐레이션 가이드
       ├─→ [Summoner Service]   소환사 검색, 매치 히스토리
       ├─→ [AI Feedback Service] 매치 분석, LLM 연동 (유료)
       └─→ [Auth Service]       회원가입, 로그인, RSO 연동, 결제

[Collector Service]  Riot API 폴링, 매치 수집, Snowball 확장
       │
       ▼ (이벤트)
[Analytics Service]  덱 클러스터링, 티어 산정, 꿀 덱 발굴, 통계 집계
       │
       ▼ (결과 저장)
[Comp DB] ← Comp Service가 읽어서 서빙
```

### 서비스별 역할

| 서비스 | 역할 | 스케일링 특성 |
| --- | --- | --- |
| API Gateway | 라우팅, JWT 검증, Rate Limiting | 트래픽 비례 수평 확장 |
| Comp Service | 추천 덱/꿀 덱 조회, 큐레이션 가이드 CRUD | 읽기 위주, 캐싱 효과 큼 |
| Summoner Service | 소환사 프로필, 매치 히스토리 조회 | Riot API 프록시 + 캐싱 |
| AI Feedback Service | 매치 분석, LLM 호출 | 유료 기능, 독립 스케일링 |
| Auth Service | 회원/인증/결제 | 낮은 트래픽, 높은 안정성 |
| Collector Service | Riot API 폴링, 매치 수집 | API Rate Limit에 맞춰 조절 |
| Analytics Service | 배치 통계, 클러스터링 | CPU 집약, 주기적 배치 |

### 서비스 분리 근거

| 분리 | 근거 |
| --- | --- |
| Collector ↔ Analytics | 수집은 I/O 바운드(API 폴링), 분석은 CPU 바운드(클러스터링) — 스케일링 패턴이 다름 |
| Comp ↔ Summoner | Comp은 내부 DB 읽기(캐싱), Summoner는 Riot API 실시간 프록시 — 장애 격리 필요 |
| AI Feedback 독립 | 유료 기능 + LLM 비용 → 독립 모니터링/스케일링, 장애가 무료 기능에 영향 X |
| Auth 독립 | 보안 민감, 변경 주기가 다름, 결제 로직 격리 |

## DB 전략

서비스별 독립 DB (Database per Service) 원칙.

| 서비스 | DB | 선택 이유 |
| --- | --- | --- |
| Collector | MySQL | 매치 원본 데이터, 시드 풀 관리 — 관계형 적합 |
| Analytics | MySQL | 집계 결과 저장, 복잡한 쿼리 |
| Comp | MySQL + Redis | 덱/가이드 데이터 + 조회 캐싱 |
| Summoner | Redis (캐시) + MySQL | Riot API 응답 캐싱 + 즐겨찾기/검색 이력 |
| AI Feedback | MySQL | 분석 결과 저장, 사용량 추적 |
| Auth | MySQL | 유저, 세션, 결제 내역 |

> 하나의 MySQL 인스턴스에 데이터베이스(schema)로 논리 분리 → 트래픽 증가 시 물리 분리

## 서비스 간 통신

### 통신 방식

| 통신 경로 | 방식 | 이유 |
| --- | --- | --- |
| API Gateway → 각 서비스 | REST (동기) | 클라이언트가 응답을 기다림 |
| Collector → Analytics | Redis Streams (비동기) | 수집 완료 후 분석은 즉시 안 해도 됨, 배치로 모아서 처리 |
| Analytics → Comp DB | 직접 DB 쓰기 | 배치 집계 결과를 Comp이 읽는 DB에 저장, 이벤트 불필요 |
| Summoner → Collector | Redis Streams (비동기) | 미수집 소환사 요청 시 큐에 넣고 즉시 반환, 수집은 백그라운드 |
| AI Feedback → Comp | REST (동기) | 분석 시점에 최신 덱 통계가 필요, 유저가 결과 대기 중 |
| AI Feedback → Summoner | REST (동기) | 매치 상세 데이터 조회 필요, 유저가 결과 대기 중 |
| Collector → Summoner | Redis Streams (비동기) | 새 매치 수집 시 소환사 캐시 갱신 알림 |

### 이벤트 스트림 (Redis Streams)

| 스트림 | 발행자 | 구독자 | 페이로드 |
| --- | --- | --- | --- |
| `match.collected` | Collector | Analytics | `{ matchId, patchVersion, region }` |
| `match.batch.ready` | Collector | Analytics | `{ patchVersion, matchCount }` — 일정 건수 누적 시 배치 트리거 |
| `summoner.collect.request` | Summoner | Collector | `{ puuid, region }` — 미수집 소환사 매치 수집 요청 |
| `stats.updated` | Analytics | Comp, Summoner | `{ patchVersion, updatedAt }` — 통계 갱신 완료 알림, 캐시 무효화 |
| `patch.detected` | Collector | Analytics, Comp | `{ newVersion, previousVersion }` — 새 패치 감지 |

### 통신 흐름도

```
[Client] → [API Gateway] ──REST──→ [Comp Service] ──캐시 조회──→ [Redis]
                          ──REST──→ [Summoner Service] ──REST──→ [Riot API]
                          │                │
                          │                └──Redis Streams──→ [Collector]
                          │                                        │
                          ──REST──→ [AI Feedback] ──REST──→ [Comp Service]
                                                  ──REST──→ [Summoner Service]

[Collector] ──Redis Streams──→ [Analytics] ──DB 쓰기──→ [Comp DB]
     │                              │
     └──Redis Streams──→ [Summoner] └──Redis Streams──→ [Comp, Summoner] (캐시 무효화)
```

## 정적 데이터 관리

| 항목 | 방식 |
| --- | --- |
| Data Dragon / Community Dragon | Collector Service가 패치 감지 시 다운로드 → S3 업로드 |
| 챔피언/아이템/증강 메타데이터 | S3 JSON + CloudFront CDN |
| 이미지 | S3 + CloudFront CDN |

## 인프라 구성 (AWS)

### 초기: EC2 올인원

EC2 한 대에 Docker Compose로 전체 서비스 + MySQL + Redis를 운영.

```
[Route 53] → [CloudFront]
                  │
        ┌─────────┴──────────┐
        ▼                    ▼
  [S3 정적 자산]      [EC2 t4g.large]
                     Docker Compose
                      ├─ API Gateway
                      ├─ Comp Service
                      ├─ Summoner Service
                      ├─ AI Feedback Service
                      ├─ Auth Service
                      ├─ Collector Service
                      ├─ Analytics Service
                      ├─ MySQL
                      └─ Redis (캐시 + Streams)
```

| 구성 요소 | 스펙 | 월 비용 (USD) |
| --- | --- | --- |
| EC2 | t4g.large (2 vCPU / 8GB) | ~$49 |
| CloudFront + S3 | 정적 자산 | ~$7 |
| Route 53 | 1개 | ~$1 |
| **합계** | | **~$57/월** |

#### 메모리 배분

| 컴포넌트 | 할당 메모리 |
| --- | --- |
| Spring Boot × 7 서비스 | ~3.5GB (각 512MB) |
| MySQL | ~1.5GB |
| Redis | ~0.5GB |
| OS + 여유분 | ~2.5GB |
| **합계** | **~8GB** |

#### 직접 운영 시 필수 작업

| 항목 | 방법 |
| --- | --- |
| MySQL 백업 | `mysqldump` 크론잡 → S3 업로드 (일 1회) |
| Redis 영속성 | `appendonly yes` (AOF) 활성화 |
| 디스크 보호 | EBS 스냅샷 주 1회 자동 생성 |
| 서비스 메모리 제한 | Docker Compose `mem_limit`으로 서비스별 상한 설정 |
| 모니터링 | CloudWatch Agent로 CPU/메모리/디스크 알림 |

### 스케일업: 인프라 분리

서비스 성장 시 단계적으로 관리형 서비스로 전환.

| 단계 | 전환 내용 | 트리거 |
| --- | --- | --- |
| 1 | MySQL → RDS | DB 부하 증가, 백업/복구 자동화 필요 |
| 2 | Redis → ElastiCache | 캐시 미스 증가, 고가용성 필요 |
| 3 | EC2 → ECS Fargate | 서비스별 독립 스케일링 필요 |
| 4 | Redis Streams → MSK (Kafka) | 이벤트 리플레이, 높은 처리량 필요 |
| 5 | ALB 추가 | 멀티 인스턴스 로드밸런싱 |
