## 초기: EC2 올인원

EC2 한 대에 Docker Compose로 전체 서비스 + MySQL + Redis를 운영.

```
[Route 53] → [CloudFront (HTTPS + CDN)]
                      │
              경로 기반 분기
              ┌───────┴───────┐
              ▼               ▼
        /static/*         /api/*
        [S3 정적 자산]    [EC2 t4g.large]
                         Docker Compose
                          ├─ API Gateway
                          ├─ Comp Service
                          ├─ Summoner Service
                          ├─ AI Feedback Service
                          ├─ Auth Service
                          ├─ Collector Service
                          ├─ Analytics Service
                          ├─ MySQL
                          └─ Redis (캐시 + Streams)
```

CloudFront가 단일 도메인(`lolchetoss.com`)에서 경로 기반으로 분기:
- `/static/*` → S3 (정적 자산, CDN 캐싱)
- `/api/*` → EC2 (백엔드, 캐싱 비활성화)
- HTTPS는 CloudFront + ACM(무료 SSL 인증서)으로 자동 처리

### 비용

| 구성 요소 | 스펙 | 월 비용 (USD) |
| --- | --- | --- |
| EC2 | t4g.large (2 vCPU / 8GB) | ~$49 |
| CloudFront + S3 | 정적 자산 + API 프록시 | ~$7 |

### 직접 운영 시 필수 작업

| 항목 | 방법 |
| --- | --- |
| MySQL 백업 | `mysqldump` 크론잡 → S3 업로드 (일 1회) |
| Redis 영속성 | `appendonly yes` (AOF) 활성화 |
| 디스크 보호 | EBS 스냅샷 주 1회 자동 생성 |
| 서비스 메모리 제한 | Docker Compose `mem_limit`으로 서비스별 상한 설정 |
| 모니터링 | CloudWatch Agent로 CPU/메모리/디스크 알림 |

## 정적 데이터 관리

### 왜 필요한가

사이트에서 보여주는 데이터는 두 종류:

| 종류 | 예시 | 변경 주기 | 출처 |
| --- | --- | --- | --- |
| 동적 데이터 | 덱 승률, 매치 히스토리, 소환사 전적 | 수시 | 우리 DB |
| 정적 데이터 | 챔피언 이름/이미지, 아이템 아이콘, 증강 아이콘 | 패치 때만 (~2주) | Riot Data Dragon |

### 왜 Riot CDN을 직접 안 쓰는가

Riot이 Data Dragon이라는 공개 CDN으로 정적 데이터를 제공하지만, 직접 쓰면 문제가 있다:

- Riot CDN 장애 시 우리 사이트 이미지가 전부 깨짐
- 응답 속도를 우리가 제어할 수 없음
- 데이터 가공 불가 (한국어 이름 매핑, 누락 데이터 보완 등)

그래서 **Riot에서 다운로드 → 우리 S3에 저장 → CloudFront로 서빙**하는 구조.

### 시나리오: 패치 14.21 → 14.22 업데이트

```
Collector가 1시간마다 Data Dragon 버전 API 확인
    ↓ "14.22 나왔다!"
Collector가 14.22 데이터 다운로드
  - Data Dragon: 챔피언/아이템/증강/특성 이미지 + 기본 정보
  - Community Dragon: Data Dragon에 없는 상세 데이터 (특성 목록, 아이템 스탯 등)
    ↓
S3에 업로드
  /static/14.22/champions/jinx.png
  /static/14.22/items/bf-sword.png
  /static/14.22/metadata.json  ← 이름, 코스트, 특성 등 통합 JSON
    ↓
CloudFront (CDN)가 S3 앞에서 전세계 캐싱 → 빠른 응답
```

### 시나리오: 유저가 덱 페이지를 볼 때

```
React 앱이 두 곳에서 데이터를 가져옴:
  ├─ 덱 통계 (승률, 등수) ← API Gateway → Comp Service (백엔드)
  └─ 챔피언 이미지, 아이템 아이콘 ← CloudFront (CDN, 백엔드 안 거침)
```

이미지 같은 정적 파일은 백엔드를 거치지 않고 CDN에서 직접 가져옴 → 백엔드 부하 절감.

### 구성 요약

| 항목 | 방식 |
| --- | --- |
| 데이터 소스 | Data Dragon + Community Dragon |
| 다운로드 주체 | Collector Service (패치 감지 시) |
| 저장소 | S3 (패치 버전별 디렉토리) |
| 서빙 | CloudFront CDN |
| 버전 관리 | `/static/{패치버전}/` 디렉토리 구조 |

## 배치 처리

### 통계 집계 (Analytics Service)

1시간마다 수집된 매치 데이터를 분석하여 덱 통계를 갱신.

```
@Scheduled (1시간 주기) 또는 match.batch.ready 이벤트 수신
    ↓
Collector API에서 최근 매치 데이터 조회
    ↓
덱 클러스터링 → 티어 산정 → 꿀 덱 발굴
    ↓
Comp API로 결과 전달
    ↓
stats.updated 이벤트 발행 (Summoner 캐시 무효화)
```

### 왜 Spring Batch를 안 쓰는가

| 기준 | 현재 상황 | Spring Batch가 필요한 시점 |
| --- | --- | --- |
| 1회 처리량 | ~3,000 매치 (Personal Key 1시간) | 수십만 건 이상 |
| 실패 복구 | 처음부터 다시 돌려도 1시간 내 완료 | 재시작에 수 시간 소요 |
| 처리 이력 | 로그로 충분 | chunk별 진행 상태 추적 필요 |

초기에는 `@Scheduled` + 서비스 로직으로 충분. Production Key(하루 ~40만 매치)로 스케일업 후 도입 검토.

## 스케일업: 인프라 분리

서비스 성장 시 단계적으로 관리형 서비스로 전환. DB/캐시가 먼저 분리되어야 EC2 수평 확장이 가능.

| 단계 | 전환 내용 | 트리거 |
| --- | --- | --- |
| 1 | MySQL → RDS | DB 부하 증가, 백업/복구 자동화 필요 |
| 2 | Redis → ElastiCache | 고가용성 필요 |
| 3 | EC2 2대 + ALB | 단일 서버로 트래픽 감당 불가 (DB/캐시 분리 후 가능) |
| 4 | EC2 → ECS Fargate | 서비스별 독립 스케일링 필요 |
| 5 | Redis Streams → MSK (Kafka) | 이벤트 리플레이, 높은 처리량 필요 |
| 6 | `@Scheduled` → Spring Batch | 1회 집계 데이터가 수십만 건 이상 |
